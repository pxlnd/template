# Codex Project Rules

Этот файл описывает базовые правила для разработки небольших web-проектов и UI-модулей, которые могут встраиваться в игру, WebView или Unity через bridge-схемы. Перед началом задачи Codex должен считать эти правила рабочим контрактом проекта.

## Главные принципы

- Пиши код так, чтобы его можно было перенести в другой проект без скрытых зависимостей.
- Сохраняй существующую архитектуру, стиль именования, форматирование и подход к DOM/CSS.
- Не делай крупные рефакторы без необходимости. Правка должна быть настолько узкой, насколько позволяет задача.
- Не ломай текущие публичные API, глобальные функции, события и схемы интеграции.
- Если задача не требует нового фреймворка, не добавляй его.
- Любая интеграция с Unity/WebView должна быть изолирована в интеграционном слое, а не внутри переиспользуемых UI-компонентов.
- При неясности выбирай самое консервативное решение, которое сохраняет обратную совместимость.

## Чистая архитектура

Проект должен быть разделен на понятные слои:

- `index.html` - точка сборки страницы, подключение скриптов, стилей и базовой разметки.
- `scripts/` - логика приложения, UI-компоненты, bridge-интеграции, debug-инструменты.
- `css/` - стили компонентов и экранов.
- `content/` - ассеты: иконки, изображения, шрифты, звуки.
- `data/` - конфиги, статические данные, тексты, настройки уровней.
- `rules/` - документация по отдельным переносимым модулям.

Рекомендуемое разделение ответственности:

- UI-компонент создает DOM, хранит локальное состояние, рендерит себя и вызывает переданные callbacks.
- Интеграционный слой создает компоненты, связывает их между собой и отправляет внешние события.
- Bridge к Unity/WebView живет рядом с boot/init-кодом, а не внутри автономного компонента.
- Debug-код не должен влиять на production-поведение.

## Подход к разработке

Перед изменением кода:

1. Найди существующий обработчик, компонент или API, который уже решает похожую задачу.
2. Проверь, нет ли документации в `rules/`.
3. Определи минимальное место изменения.
4. Проверь, не является ли файл переиспользуемым модулем, который не должен знать о конкретном проекте.

Во время реализации:

- Используй существующие helpers и паттерны проекта.
- Для событий используй callbacks компонента, а внешние переходы отправляй из интеграционного слоя.
- Для URL-схем используй `encodeURIComponent` для параметров.
- Не дублируй состояние, если его можно получить через `getState()`.
- Не добавляй глобальные переменные без необходимости.
- Не смешивай изменение данных, рендер и внешнюю навигацию в одной функции, если это можно аккуратно разделить.

После реализации:

- Проверь синтаксис измененных JS-файлов, например `node --check scripts/main.js`.
- Проверь `git diff`, чтобы убедиться, что правка не затронула лишние файлы.
- Если менялась верстка, проверь внешний вид в браузере на desktop и mobile размерах.

## Планирование задач

Для простой задачи:

- Найди нужный файл.
- Внеси минимальную правку.
- Проверь синтаксис или поведение.
- Кратко опиши результат.

Для сложной задачи:

1. Сформулируй цель и пользовательский сценарий.
2. Раздели работу на маленькие шаги.
3. Определи файлы, которые будут затронуты.
4. Сначала сделай основу поведения, затем визуальные детали, затем edge cases.
5. В конце проверь интеграцию целиком.

Не начинай с большого переписывания. Сначала докажи, что изменение можно встроить в текущую систему.

## Структура компонентов

Автономный UI-компонент должен:

- принимать настройки через `options`;
- иметь локальное `state`;
- предоставлять методы `show()`, `hide()`, `destroy()`, `getState()` там, где это уместно;
- отдавать наружу события через callbacks: `onClose`, `onRestart`, `onStateChange` и т.п.;
- не вызывать напрямую Unity/WebView-схемы, если компонент предполагается переносимым;
- сам подключать свой CSS только если это уже принято в проекте;
- использовать CSS-классы с уникальным префиксом компонента.

Компонент не должен:

- знать о других экранах без явной необходимости;
- читать или менять глобальное состояние напрямую;
- отправлять analytics/track события самостоятельно, если это можно сделать в integration layer;
- содержать debug-логи в production-сценариях.

## Unity/WebView bridge

Схемы вида `uniwebview://...` считаются внешней интеграцией.

Правила:

- Отправляй их из `main.js` или другого integration/bootstrap слоя.
- Не встраивай `uniwebview://...` в переиспользуемые компоненты.
- Параметры URL всегда кодируй через `encodeURIComponent`.
- Если нужно отправить несколько событий подряд, учитывай, что смена `location.href` может прервать предыдущий переход. При необходимости используй небольшую задержку между событиями.
- Не меняй существующие схемы без явного запроса.

Примеры:

```js
function trackUnityEvent(event, action) {
  var eventName = encodeURIComponent(String(event));
  var eventAction = encodeURIComponent(String(action));
  window.location.href = 'uniwebview://track?event=' + eventName + '&event_action=' + eventAction;
}
```

```js
function navigateClose(state) {
  var coins = encodeURIComponent(String(state.coinsCount));
  var hearts = encodeURIComponent(String(state.heartsCount));
  window.location.href = 'uniwebview://close?coins=' + coins + '&hearts=' + hearts;
}
```

## JavaScript правила

- Сохраняй совместимость с простым браузерным JS, если проект уже написан без сборщика.
- Используй IIFE или существующий модульный паттерн проекта.
- Используй `var`, если весь файл уже написан в старом стиле.
- Не добавляй TypeScript, Babel, npm-зависимости или сборку без явной необходимости.
- Не используй современные API без проверки совместимости с целевой WebView-средой.
- Обработчики событий должны быть простыми и делегировать сложную логику в отдельные функции.
- Функции должны называться по действию: `navigateRestart`, `trackUnityEvent`, `bindScreenTapTracking`.

## CSS правила

- Каждый экран или компонент должен иметь собственный CSS-файл.
- Используй уникальный префикс классов компонента: например `ls-`, `mh-`, `qs-`.
- Не вводи глобальные стили, если можно ограничиться компонентом.
- Стили должны быть устойчивы к разным размерам экрана.
- Не меняй размеры, отступы и позиционирование соседних элементов без необходимости.
- Для скрытия элементов используй уже существующий подход проекта.
- Ассеты подключай через CSS custom properties или через существующий asset path, если это уже используется.

## Ассеты

- Храни игровые и UI-ассеты в `content/`.
- Не переименовывай существующие ассеты без необходимости.
- Не заменяй изображения, если задача не про визуальный редизайн.
- Если добавляешь ассет, клади его в тематическую папку: `content/icons/...`, `content/images/...`, `content/sounds/...`.
- Пути к ассетам должны быть относительными и переносимыми.

## Работа с состоянием

- Состояние компонента должно быть внутри компонента.
- Общая синхронизация между экранами должна быть в integration layer.
- После изменения state обязательно обновляй UI и отправляй `onStateChange`, если такой callback есть.
- Не полагайся на DOM как на источник данных, если состояние уже хранится в объекте.
- Не изменяй state извне напрямую, если есть публичный метод.

## События и взаимодействия

- Клики по кнопкам должны вызывать callbacks компонента.
- Глобальные события страницы нужно фильтровать, чтобы они не перехватывали клики по интерактивным элементам.
- Не отправляй один и тот же analytics event дважды из разных слоев.
- Если событие нужно для Unity, называй его и action ровно так, как указано в задаче.
- При изменении поведения кнопок сохраняй старую навигацию, если пользователь не просит убрать ее.

## Debug и тестирование

- Debug-панель и hotkeys должны оставаться вспомогательными инструментами.
- Debug-кнопки не должны отправлять production analytics, если это не оговорено.
- Минимальная проверка для JS-правок: `node --check path/to/file.js`.
- Для UI-правок проверяй:
  - desktop viewport;
  - mobile viewport;
  - отсутствие overlap;
  - клики по основным кнопкам;
  - видимость нужных экранов.

## Git и безопасность изменений

- Перед правкой проверь `git status --short`.
- Не откатывай чужие изменения.
- Не запускай destructive-команды вроде `git reset --hard` без явной просьбы.
- Не форматируй весь файл, если меняешь одну функцию.
- В финальном ответе укажи, какие файлы изменены и какая проверка выполнена.

## Качество результата

Хорошая правка:

- решает конкретную задачу;
- не меняет лишнее поведение;
- легко читается;
- следует локальному стилю проекта;
- проверена минимальным релевантным способом;
- оставляет проект в состоянии, где следующему разработчику понятно, куда смотреть.

Плохая правка:

- смешивает UI, бизнес-логику и интеграции;
- добавляет новый паттерн без причины;
- прячет важную логику в CSS или глобальных side effects;
- ломает переносимость компонента;
- не проверяет базовый сценарий после изменения.
